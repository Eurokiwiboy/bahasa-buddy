Design Section 3: Supabase CLI & Schema Migrations

Version-controlling your database so you never lose track of what's deployed and can reproduce your setup anywhere.

Setup

New project structure (supabase/ directory)
supabase/
  ├── config.toml ← updated with your project ID
  ├── migrations/
  │   ├── 00000000000000_baseline.sql ← exported current schema
  │   ├── 00000000000001_enable_rls.sql ← RLS policies from Section 2
  │   └── 00000000000002_secure_rpc.sql ← lock down add_xp function
  └── seed.sql ← optional: sample categories/cards for fresh setup

How It Works

  • Baseline migration — we'll export your current live schema using supabase db dump and save it as the first migration. This captures all your existing tables exactly as they are.
  • RLS migration — the policies from Section 2 go in a separate migration file so they're trackable and reversible.
  • Future changes — any new tables, columns, or policies get their own migration file. Run supabase db push to apply.
  • config.toml — updated to point at your project (zxmwfv...), removing all Lovable references.

Developer Workflow Going Forward

Commands you'll use
$ supabase link --project-ref zxmwfvyqrtqtsrfhdvhv
# Links local project to your remote Supabase project
$ supabase db diff --schema public
# Shows differences between local migrations and remote DB
$ supabase migration new add_some_feature
# Creates a new timestamped migration file
$ supabase db push
# Applies pending migrations to your remote database

📦 Prerequisites

You'll need to install the Supabase CLI (brew install supabase/tap/supabase on macOS) and log in with supabase login. We'll include these steps in the implementation plan.
