Security & Infrastructure: 3 Approaches

All approaches fix the exposed keys and missing RLS. They differ in how much infrastructure we set up now vs later.

A

🩹 Minimal Patch (fastest)

Do: Enable RLS on all tables, rotate anon key, move credentials to .env, add .env to .gitignore, create .env.example, scrub git history of keys.

Skip: No CLI setup, no migrations, no schema versioning.

Effort: ~2 hours  |  Risk after: Low (keys secured, RLS active)  |  Tech debt: High (no schema versioning)
B

🏗️ Proper Foundation (recommended)

Do: Everything in A, plus: set up Supabase CLI, export current schema as baseline migration, add RLS policies as code, add .env.example, update README, add basic tests for auth flow.

Skip: No CI/CD pipeline, no staging environment.

Effort: ~4-6 hours  |  Risk after: Very low  |  Tech debt: Low (schema in code, reproducible)
C

🚀 Production-Grade (most thorough)

Do: Everything in B, plus: separate staging/prod Supabase projects, GitHub Actions CI, automated migration checks, Supabase Edge Functions for sensitive operations, security headers, CSP policy.

Effort: ~2-3 days  |  Risk after: Minimal  |  Tech debt: Very low (production-ready infra)

💡 My Recommendation: Option B

It fixes the urgent security holes AND gives you a proper development foundation (migrations, env handling, schema-as-code) without overengineering for a personal project. Option C makes sense later when you're ready for SaaS, but it's overkill right now.
