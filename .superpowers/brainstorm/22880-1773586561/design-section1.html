Design Section 1: Credential Security

The most urgent piece — securing exposed API keys and preventing future leaks.

What Changes

Before (current state)
❌ .env committed to repo with Lovable's Supabase keys
❌ client.ts has YOUR keys hardcoded, overriding .env
❌ Both sets of keys visible in git history
❌ No .gitignore entry for .env
❌ No .env.example for new contributors
After (proposed)
✅ .env added to .gitignore (never committed again)
✅ client.ts reads from import.meta.env (no hardcoded keys)
✅ .env.example with placeholder values committed
✅ Supabase anon key rotated in dashboard
✅ Git history scrubbed of old keys (BFG Repo-Cleaner)
✅ Lovable project references fully removed

Key Decisions

  • Rotate the anon key in Supabase dashboard — the old one is compromised
  • client.ts will use import.meta.env.VITE_SUPABASE_URL and import.meta.env.VITE_SUPABASE_ANON_KEY
  • Git history scrub — use BFG Repo-Cleaner to remove old .env and hardcoded keys from all past commits
  • Lovable deployment — update env vars in Lovable's deployment settings after key rotation

⚠️ Note: Manual Steps Required

Key rotation and Lovable env var updates must be done by you in the Supabase dashboard and Lovable settings. We'll provide exact instructions in the plan.
