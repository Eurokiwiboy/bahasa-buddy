Design Section 2: Row Level Security (RLS)

Locking down who can read and write what in your database. This is your primary defense even when the anon key is public (which is normal for Supabase).

RLS Policy Design Per Table

User-Owned Data (strict — users see only their own)
profiles — SELECT/UPDATE own row only (auth.uid() = id)
user_card_progress — SELECT/INSERT/UPDATE own rows only
user_lesson_progress — SELECT/INSERT/UPDATE own rows only
daily_goals — SELECT/INSERT/UPDATE own rows only
user_achievements — SELECT/INSERT own rows only
xp_transactions — SELECT own rows only, INSERT via RPC only
Shared Content (read-only for authenticated users)
categories — SELECT for all authenticated users, no INSERT/UPDATE/DELETE
splash_cards — SELECT for all authenticated users, no INSERT/UPDATE/DELETE
lessons — SELECT for all authenticated users, no INSERT/UPDATE/DELETE
phrases — SELECT for all authenticated users, no INSERT/UPDATE/DELETE
achievements — SELECT for all authenticated users, no INSERT/UPDATE/DELETE
Community Chat (scoped to room membership)
chat_rooms — SELECT for all authenticated users
chat_room_members — SELECT/INSERT own membership, DELETE own membership
chat_messages — SELECT if member of room, INSERT if member of room, UPDATE/DELETE own messages only
message_reactions — SELECT if member of room, INSERT/DELETE own reactions only

Key Principles

  • Default deny — RLS enabled on all tables, no policies = no access
  • Content tables are read-only — only admins (via service_role key in dashboard) can add lessons/cards
  • Chat scoped to membership — must be a room member to see/send messages
  • XP insert via RPC only — prevents users from giving themselves arbitrary XP
  • All policies written as migration files — version controlled and reproducible
